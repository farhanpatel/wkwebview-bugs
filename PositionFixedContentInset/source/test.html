Position: fixed content inset bug

Tap the number in any row to dynamically add a new element, which is configured to overlay the current element exactly. This is positioned using `position: fixed` and the rect obtained from tapped element by calling `element.getBoundingClientRect()`. When the content insets are controlled manually, `position: fixed` elements are not positioned correctly. This succeeds in tab 1, but fails in tab 2. This can be confirmed visually and by comparing the `overlay.getBoundingClientRect()` to the `overly.style.top`. Ensure you scroll at least a little, the bug doesn't occur when the content is already at the top.

In tabs 3 and 4, it succeeds. They are the same as 1 and 2, but the view doesn't sit beneath the nav bar.
